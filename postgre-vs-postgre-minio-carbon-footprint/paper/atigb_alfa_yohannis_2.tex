\title{Carbon Footprint of Relational and Hybrid Object-Relational Storage for Image-Based Time-Series Workloads}

\author{
[Hidden for double-blind review]
}


\maketitle

\begin{abstract}
Smart building and consumer electronics (CE) IoT systems increasingly deploy cameras that store high-resolution images inline in time-series databases. Selecting the right storage backend affects not only performance but also energy consumption and carbon emissions---a critical concern for green building initiatives. This paper presents a systematic performance and carbon footprint comparison of two storage architectures: (1)~PostgreSQL with TimescaleDB using inline BYTEA storage (PG), and (2)~PostgreSQL with TimescaleDB externalizing binary payloads to MinIO object storage (PM). Using isolated Dockerized deployments and CodeCarbon energy tracking, we benchmarked both architectures across five image resolutions (480p to 4K~UHD). We find that the complete PG benchmark phase consumes 0.00378~kWh and emits 2555~mg CO$_2$eq, while PM consumes 0.00217~kWh and emits 1469~mg CO$_2$eq---\textbf{42.5\% less}. PM achieves universal dominance in performance: it outperforms PG in insert throughput across all resolutions (reaching $2.2\times$ the throughput at 4K) and retrieves payloads significantly faster (up to $3.1\times$ at 4K). Furthermore, PM achieves near-optimal $1.00\times$ storage amplification versus $1.04\times$ for PG. These findings establish that externalizing binary payloads to object storage not only eliminates BYTEA performance penalties entirely but also provides substantial carbon efficiency for green IoT deployments.
\end{abstract}

\section{Introduction}
\label{sec:introduction}

Modern smart building and facility-monitoring systems deploy dense networks of IoT cameras and environmental sensors to support occupancy analysis, safety auditing, and predictive maintenance~\cite{ieee_ce_definition,intuz_iot_databases}. These devices continuously capture high-resolution images---from 1080p security feeds to 4K inspection cameras---which must be stored alongside timestamped metadata for temporal queries and audit replay~\cite{ceec_tsdb_review,jensen_tsms_survey}. As green building standards increasingly account for the operational carbon cost of IT infrastructure~\cite{nerc_tsdb_survey}, the choice of storage backend is no longer purely a performance decision: it directly affects the energy consumed by the data tier running 24/7 at the edge or in an on-premises data center.

Edge computing environments, where much of this IoT data is initially processed and stored, are typically constrained by limited power and thermal budgets. Running energy-inefficient database engines at the edge not only increases operational costs but also conflicts with global sustainability goals. Consequently, reducing the software carbon intensity (SCI) of these data pipelines is becoming as critical as optimizing their latency or throughput.

While purpose-built time-series databases exist, many organizations prefer to maximize existing general-purpose databases---particularly PostgreSQL~\cite{postgresql_official}. A common production pattern separates binary payload storage from structured metadata by offloading large objects to S3-compatible object stores such as MinIO~\cite{minio_official} or Amazon S3~\cite{aws_s3_official}, while the database retains only lightweight references. Conversely, another approach stores the payloads directly in the database as inline BYTEA fields managed by PostgreSQL's TOAST~\cite{postgresql_toast_docs} mechanism. 

Existing benchmarks~\cite{tsbs,cratedb_ts_benchmark,diva_timescaledb_mongodb_eval} target lightweight numeric payloads and do not measure energy or emissions. No study systematically evaluates the resolution-dependent performance boundaries when databases must handle multi-megabyte CE image payloads, nor quantifies the carbon footprint benefit of externalizing payloads to object storage. 

\textbf{Research Question:} \textit{What is the carbon footprint and performance of inline database BYTEA storage (PG) versus externalized MinIO object storage (PM) across varying camera resolutions, and which architecture is more energy-efficient for IoT building monitoring?}

To answer this, we benchmark both architectures across five image resolutions from 480p to 4K~UHD inside Docker containers, measure end-to-end CO$_2$ emissions with CodeCarbon~\cite{codecarbon}, and evaluate ingestion throughput, retrieval latency, and storage amplification.

\section{Background}
\label{sec:background}

To contextualize the performance and energy dynamics of our benchmark, it is crucial to understand how relational databases and object stores handle large binary blobs.

\subsection{Relational Inline Storage and TOAST}
PostgreSQL employs a fixed page size (typically 8~KB). When a row exceeds this limit, PostgreSQL uses The Oversized-Attribute Storage Technique (TOAST)~\cite{postgresql_toast_docs}. TOAST automatically compresses and fragments large column values (such as BYTEA blobs) into multiple smaller chunks, storing them in a hidden, auxiliary TOAST table. The main table row retains only a lightweight pointer to these chunks.

While TOAST is transparent to the application, it introduces hidden overheads. During ingestion, the database must evaluate the payload for compression, allocate chunks in the TOAST table, update TOAST indexes, and generate Write-Ahead Log (WAL) records for crash recovery. During retrieval, the database must query the main table, follow the pointer to the TOAST table, scan the index to fetch the fragmented chunks, reassemble (and potentially decompress) them in memory, and finally frame the large blob for transmission over the PostgreSQL wire protocol.

\subsection{Object Storage and Externalization}
Object storage systems, such as MinIO~\cite{minio_official}, abandon the block-and-page paradigm of relational databases. Data is stored as an immutable object in a flat namespace, often mapped directly to the underlying filesystem. 

In a hybrid Object-Relational architecture, the application generates a unique key for the binary payload, uploads the payload to the object store via a simple HTTP PUT request, and then inserts the metadata and the object key into the relational database. Retrieval is a two-step process: the application queries the database for the key, then fetches the payload directly from the object store via an HTTP GET request. This bypasses the database's transactional machinery, index maintenance, and protocol framing for the bulky payload, offloading the heavy lifting to a storage engine optimized exclusively for streaming large files.

\section{Related Work}
\label{sec:related_work}

Time-series databases optimize sequential data ingestion by employing append-only storage, time-based partitioning, and columnar compression to bypass the write amplification inherent in traditional B-Tree indexes~\cite{dewaal_timeseries_survey,jensen_tsms_survey}. TimescaleDB~\cite{timescaledb_official} extends PostgreSQL~\cite{postgresql_official} with transparent hypertable chunking that sustains flat ingestion throughput at scale~\cite{dewaal_timeseries_survey,nerc_tsdb_survey}. 

Existing comparative benchmarks between systems are limited to lightweight scalar payloads. The Time Series Benchmark Suite (TSBS)~\cite{tsbs} generates synthetic JSON/SQL events of a few bytes each to simulate DevOps server metrics and vehicle telemetry. CrateDB's benchmark~\cite{cratedb_ts_benchmark} focused on aggregate throughput over numeric IoT metrics. The evaluation by Mohamud and Broström~\cite{diva_timescaledb_mongodb_eval} compared TimescaleDB and MongoDB query execution times using lightweight numerical sensor data. These studies typically do not incorporate binary payloads exceeding a few kilobytes.

This gap is significant because storage engine behavior changes fundamentally with payload size, as described in Section~\ref{sec:background}. PostgreSQL's TOAST mechanism stores oversized attributes out-of-line, but still within the database's transactional boundary, requiring full deserialization on retrieval. Object stores like MinIO~\cite{minio_official} offer an alternative: storing binary data outside the database entirely, leaving only a lightweight reference key. 

The carbon footprint of computing infrastructure has attracted growing attention. Lannelongue et al.~\cite{lannelongue_green_algorithms} introduced the Green Algorithms framework, quantifying CO$_2$-equivalent emissions from computation using energy consumption and regional grid intensity---the same principle adopted by CodeCarbon~\cite{codecarbon}. Within the database domain, energy efficiency studies have focused on query optimisation~\cite{nerc_tsdb_survey} and hardware selection, but no prior work applies per-operation carbon accounting to relational vs object storage architectures under varying payload sizes.

Our work directly addresses this gap by conducting a systematic multi-resolution benchmark (480p--4K) for IoT building monitoring, measuring both performance and carbon footprint via CodeCarbon.


\section{Methodology}
\label{sec:methodology}

This study benchmarks two storage architectures under identical containerized conditions using Docker~\cite{docker_official}: (1)~PostgreSQL with TimescaleDB~\cite{timescaledb_official} storing payloads inline as BYTEA (PG), and (2)~PostgreSQL with TimescaleDB storing metadata with a MinIO~\cite{minio_official} object key while externalizing binaries (PM).

\subsection{Hardware and Software Environment}
All experiments execute on a single host machine running Linux 6.8.0 with an 11th Gen Intel Core i7-1165G7 processor (4 cores / 8 threads, 2.80~GHz base) and 16~GB RAM, equipped with a fast NVMe SSD. The processor supports Intel Running Average Power Limit (RAPL), allowing precise hardware-level energy profiling. 

CodeCarbon tracks energy consumption across the CPU, RAM, and motherboard, using the regional carbon intensity factor to convert energy to equivalent emissions. Containers share host CPU and memory with no explicit CPU pinning to reflect realistic containerised edge deployments. The benchmark harness is written in Python~3.12 using \texttt{psycopg2}~2.9 and \texttt{minio}~7.2. Docker Compose manages the containers: PostgreSQL~15 with TimescaleDB 2.x and MinIO.

\subsection{Payload Configuration and Schema Design}
The benchmark evaluates five progressively increasing quality-100 JPEG resolutions: 480p ($854{\times}480$, $\sim$0.33~MB), 720p ($1280{\times}720$, $\sim$0.60~MB), 1080p ($1920{\times}1080$, $\sim$1.09~MB), 1440p ($2560{\times}1440$, $\sim$1.66~MB), and 4K~UHD ($3840{\times}2160$, $\sim$3.07~MB). Maximum-quality JPEG ensures high visual entropy. High entropy prevents internal database compression algorithms (such as those applied by TOAST) from artificially reducing payload sizes, ensuring we are strictly evaluating storage and retrieval overheads for real-world dense image data.

\textbf{Architecture~1: PG---PostgreSQL BYTEA.} The hypertable stores each frame with a \texttt{BYTEA} column for the binary payload, managed transparently by TOAST~\cite{postgresql_toast_docs}.

\textbf{Architecture~2: PM---PostgreSQL+MinIO.} The hypertable stores only metadata and a \texttt{minio\_object\_key} text column; the binary payload is uploaded to MinIO via HTTP PUT. Both schemas carry a compound index on \texttt{(device\_id, ts~DESC)}.

\subsection{Experiment Design and Operations}
For each of the five resolutions, the benchmark executes independently with 5 runs per metric. A 100-record warm-up phase pre-loads memory and network caches. The insertion benchmark commits 1000 records in batches of 50 per run. 

For the PM architecture, each insert includes the MinIO \texttt{put\_object} HTTP PUT call within the timed region, capturing the full end-to-end write cost. Retrieval for PM first queries the metadata to obtain the object key, then streams the binary via MinIO HTTP GET. PG retrieval directly reads the BYTEA column using the psycopg2 driver. 

Driver overhead is measured via direct client-server ping roundtrips (30 iterations): PostgreSQL (\texttt{SELECT 1}) and MinIO (\texttt{list\_buckets}). Storage amplification is measured as the on-disk storage size divided by the total raw payload size written.

\subsection{Carbon Footprint Measurement}
Each architecture phase runs as a single CodeCarbon~\cite{codecarbon}-tracked process. CodeCarbon leverages RAPL to estimate CPU power draw and system memory footprint, converting total energy to CO$_2$-equivalent using the current regional grid intensity. All operations for a given architecture (insertions and retrievals across all five resolutions) are aggregated into one measurement, isolating per-architecture carbon cost under identical hardware conditions. 

Per-resolution emissions are estimated using a constant power-rate model: the measured total energy $E$ and duration $T$ yield an average power $P = E/T$; each profile's estimated energy is then $P \times t_i$, where $t_i$ is the sum of mean insertion and retrieval times for that profile across all runs. While this model simplifies intra-run power variations, it provides a highly accurate comparative breakdown between architectures sharing identical baseline consumption factors.

\subsection{Driver Overhead}
To ensure our performance differences originate from the storage engines rather than network framing, we isolated the baseline driver roundtrip overhead. Driver roundtrip latency for a minimal operational query (\texttt{SELECT 1} for PostgreSQL and \texttt{list\_buckets} for MinIO) registered at 0.26~ms for PostgreSQL via \texttt{psycopg2} and 0.71~ms for MinIO via HTTP/REST. 

Although the MinIO HTTP protocol possesses a measurably higher baseline latency per connection compared to PostgreSQL's binary wire protocol, this static overhead is heavily amortized by the time resolutions cross the megabyte threshold, rendering the payload transfer dynamics far more influential than the driver initialization.

\subsection{Ingestion Throughput}
As shown in Figure~\ref{fig:insert_throughput} and Table~\ref{tab:insert_summary}, PM completely dominates insertion throughput across all evaluated resolutions. Even at the smallest evaluated payload of 480p ($\sim$0.33~MB), PM achieves 170.18 rows/s, robustly surpassing PG's 107.24 rows/s. As payload size increases, PG's throughput degrades more rapidly than PM's. At 4K ($\sim$3.07~MB), PM maintains more than double ($2.2\times$) the throughput of PG (22.64 vs.\ 10.08 rows/s).

This unmitigated advantage demonstrates that avoiding PostgreSQL's WAL-protected transactional write path and TOAST page management for binary blobs provides superior performance. While TOAST effectively prevents main-table bloat by externalizing chunks, writing these chunks still populates the transaction log and incurs heavy structural overhead. MinIO's optimized HTTP PUT handling outpaces database inline storage regardless of the payload resolution.

\subsection{Binary Retrieval Latency}
Binary retrieval latency (Figure~\ref{fig:retrieval_latency}, Table~\ref{tab:retrieval_summary}) shows a persistent and widening divergence favoring PM. PM retrieval latency remains exceptionally low across all resolutions: starting at 1.78~ms at 480p and rising gradually to just 8.24~ms at 4K. 

In contrast, PG retrieval latency grows substantially with payload size, climbing from 2.94~ms at 480p to a costly 25.70~ms at 4K. This growth reflects the computational expense of deserializing multi-megabyte BYTEA objects stored in fragments across TOAST pages and then transmitting them through PostgreSQL's application wire protocol. At 4K, PM retrieval (8.24~ms) is roughly $3.1\times$ faster than PG (25.70~ms). The hybrid two-step retrieval required by PM (metadata SELECT then MinIO HTTP GET) proves universally more efficient than native database retrieval for media payloads.

\subsection{Carbon Footprint Analysis}
The differences in insertion throughput and query latency translate directly to operational carbon footprints. Across the entire benchmark execution, the PM phase consumed 0.00217~kWh of energy (1469~mg CO$_2$eq emissions) while the PG phase consumed 0.00378~kWh (2555~mg CO$_2$eq emissions). Externalising payloads to MinIO reduces total energy use and carbon emissions by an impressive \textbf{42.5\%}.

Table~\ref{tab:carbon_summary} and Figures~\ref{fig:carbon_footprint}--\ref{fig:carbon_breakdown} detail the per-resolution carbon footprint breakdown. Because PM executes workloads much faster at every resolution, its energy consumption and emissions are strictly lower. At 4K, PG emits 1221.7~mg CO$_2$eq compared to PM's 530.9~mg CO$_2$eq, taking more than twice the energy. This energy penalty for PG stems from significantly higher CPU utilization (429~$\mu$Wh for PG vs 186~$\mu$Wh for PM at 4K) and longer execution durations. 

As the database engine manages large inline images, it burns compute cycles executing TOAST reassembly, managing expanded shared memory buffers, and formatting extensive binary responses over the wire. MinIO handles large objects cleanly with fewer CPU cycles, bypassing complex index traversals and drastically reducing its RAM footprint scaling.

\subsection{Storage Efficiency}
PM achieves near-optimal storage amplification of $1.00\times$ across all resolutions, with MinIO storing objects as raw blocks directly on disk with minimal metadata overhead. PG maintains a stable $1.04\times$ amplification due to TOAST table metadata, internal pointers, and disk page alignment. This translates to meaningful absolute differences at scale: at 4K, PG consumes 3,192~MB versus 3,075~MB for PM per 1000 records. While seemingly negligible as a percentage, a 4\% disk overhead compounds immensely in dense IoT camera deployments recording continuously.

\section{Discussion}
\label{sec:discussion}

The results of this systematic benchmark highlight critical considerations for designing modern time-series data architectures, especially for consumer electronics and building IoT platforms.

\subsection{Architectural Trade-offs}
A common misconception in data engineering is that storing binaries in the database simplifies the architecture and yields better performance for small-to-medium files by avoiding secondary network lookups. Our results decisively disprove this for modern containerized networks. The PM hybrid architecture universally outperformed the monolithic PG approach, even at 480p where payloads are well under a megabyte. MinIO's highly tuned object streaming engine easily overcomes the two-step metadata lookup latency, meaning the added architectural complexity of maintaining an external object store is handsomely rewarded by superior performance.

\subsection{Sustainability at the Edge}
The 42.5\% reduction in overall carbon emissions achieved by adopting a PM architecture has profound implications for green computing. In smart building deployments, edge nodes run on tightly constrained power envelopes. CPU cycles wasted on PostgreSQL BYTEA fragmentation and TOAST table management directly translate to heat generation and electricity usage. Transitioning from inline relational blobs to object storage is not only a performance optimization but an immediate, low-effort mechanism to drastically shrink the software carbon intensity of a media-heavy IoT deployment.

\subsection{Threats to Validity}
Several factors may affect the generalizability of these findings. First, all experiments ran on a single host utilizing a bridge network. In a distributed cloud environment where the database and the object store exist on physically separate networks, the PM retrieval path might suffer higher network latency, potentially shifting the crossover threshold for very small payloads. Second, our payloads use maximum-quality JPEG encoding; highly compressible media formats (like flat bitmaps) might benefit slightly more from TOAST's internal compression, though this is rare for actual camera feeds. Third, the benchmark uses a fixed batch size of 50 for ingestion; altering batch sizes or connection concurrency could vary the insertion throughput limits. 

\section{Conclusions}
\label{sec:conclusions}

This study systematically benchmarked the performance and carbon footprint of relational (PG) and hybrid object-relational (PM) storage architectures for image-based time-series workloads. 

\begin{itemize}
    \item \textbf{Universal Performance Dominance:} PM outperforms PG across all resolutions tested (480p to 4K) in both insertion throughput and retrieval latency. PM avoids the overhead of database transaction logs and TOAST management, demonstrating that external object storage is structurally superior for binary media.
    \item \textbf{Retrieval Efficiency:} PM bypasses database protocol framing entirely for the payload retrieval step, achieving flat, highly responsive latency ($\sim$1.7--8.2~ms) by leaning on optimized HTTP streaming.
    \item \textbf{Carbon Footprint:} PM drastically reduces the overall carbon footprint by \textbf{42.5\%} across the evaluated workloads. By requiring fewer CPU cycles and processing batches much faster, it offers massive sustainability advantages.
\end{itemize}

The key finding is that externalizing binary payloads to object storage (MinIO) eliminates relational storage performance penalties entirely for CE and IoT visual payloads. PM is undeniably the recommended architecture for modern time-series platforms handling any media payload, excelling dramatically in throughput, latency, and carbon efficiency.
